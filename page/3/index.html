parent标签

此说法改自链接

当引入parent标签后,再引入的依赖的版本,由parent标签中坐标指向的pom来管理,当且仅当parent中的pom管理着此依赖

情景一:用于本项目的依赖管理

我的一个project,专门在project目录下的pom.xml中写入dependencyManagement标签,往其中加入依赖并指定版本.在这里管理着所有用到这些依赖的module所使用依赖的version

那么此时,其他多个module在使用一些共同的依赖如A时,可以在此module下的pom.xml中引入parent标签填入project下的pom.xml的坐标,这样在中引入依赖A时,只需填入坐标,不需要指定版本,版本已经由parent的dependencyManagement中指定了.

当我想更新依赖A的version时,只需要在project下的pom.xml中改依赖A的version,不必再去一个个module中改掉

情景二:用于引用maven库中的一些项目

比如我用spring-boot-starter-web依赖,为了方便,我在parent中写入spring-boot-starter-parent坐标

这一切的前提是,我知道spring-boot-starter-parent管理着spring-boot-starter-web依赖相应依赖的版本

此时我也只需要写spring-boot-starter-parent的version,而不需要写spring-boot-starter-web的version,因为在spring-boot-starter-parent的dependencyManagement标签中指定了spring-boot-starter-web的version

properties标签中java.version和maven.compiler.source/target的区别

都是8,所以两者是等价的,java.version更常用于springboot的pom.xml模板中

index

传入的两个参数Bbox和minResolutionInMeter,

首先获取最小的纬度值,纬度值越小,对应的地球周长越大

我们计算这篇区域的最大长度,因为地球是椭圆的,由Bbox决定的区域,不是单纯的矩形.我们先计算两个纬度之间的距离,再计算两个经度之间的最大距离(维度越小,两经度之间距离越大)

因此由Bbox计算出可以分解成出多少个小块,

以下内容参考翻译自:Damn Cool Algorithms: Spatial indexing with Quadtrees and Hilbert Curves

quadtree:

结构:从根节点每个节点有四个子节点,一直到叶子节点停止.换言之把空间地图四分再四分到一定程度(比如该区域没有索引点了).每个内部节点表示一个大框框覆盖某一片区域不存储任何索引点信息,而叶子节点存储多个索引点即地图中的实体点.

插入索引点数据:从根开始根据此点所在象限,进入对应根节点的子节点,然后按照这个规律再找下一个象限一直递归到叶子节点,然后插入.如果该叶子节点容纳索引点数目超标,则拆分叶子节点,原叶子节点变成内部节点,分化出四个叶子节点,并将索引点正确转移到对应叶子节点中.

查询:从根节点开始查每个子节点,并检查是否于要查询区域相交.如果是,则递归到该子节点.当到达叶子节点时,查询是否存在查找目标

从上可以看出quadtree是一个trie前缀树或称字典树,前缀树的特点是我们可以对其进行编码,这就引出了geohash

geohash:

geohash有quadtree的灵魂,却抛去了quadtree的树结构.quadtree叶子节点并不在同一深度,为了编码geohash保证所有叶子节点再同一深度即编码bit数目保持一致.

构造:当我们当我们抛弃树结构开始,我们就不能动态的构建搜索集,只能提前就构造好搜索集,即提前确定好深度.

查询:如果只查询一个点的话,我个人觉得和quadtree没区别

从graphhopper学多线程编程模式

2021-11-04

前提:最近在学graphhopper,看源码的看到有关多线程的部分,所以在这里做一下笔记

多线程基础

快速复习使用，详细看java核心技术第12章

线程的创建和使用

做个简单的比喻：进程=火车，线程=车厢 比喻细节见此

Java语言的JVM允许程序运行多个线程，它通过java.lang.Thread 类来体现。

1.每个线程都是通过某个特定Thread对象的run()方法来完成操作的，经常 把run()方法的主体称为线程体

2.线程的启动是通过调用该Thread对象的start()方法，而非直接调用run()

jdk5.0 之前创建新线程的两种方法

1.继承Thread类的方式

  1. 定义子类继承Thread类
  2. 子类中重写Thread类中的run方法
  3. 创建Thread子类对象,即创建了线程对象
  4. 调用线程对象start方法,start方法会为我们做好准备并调用run方法

注意:想启动多线程,必须调用start方法

2.实现Runnable接口(更建议使用)

  1. 定义子类，实现Runnable接口

  2. 子类中重写Runnable接口中的run方法

  3. 通过Thread类含参构造器创建线程对象

  4. 将Runnable接口的子类对象作为实际参数传递给Thread类的构造器中

  5. 调用Thread类的start方法：开启线程，调用Runnable子类接口的run方法

jdk5.0之后的两种新建线程的方法

Callable接口的方式

对比Runnable接口:

1.相比run()方法，可以有返回值

2.方法可以抛出异常

3.支持泛型的返回值

4.需要借助FutureTask类，比如获取返回结果

利用Future接口,对Callable接口功能的补充:

1.可以对具体Runnable、Callable任务的执行结果进行取消、查询是 否完成、获取结果等

2.FutrueTask是Futrue接口的唯一的实现类

3.FutureTask 同时实现了Runnable, Future接口。它既可以作为 Runnable被线程执行，又可以作为Future得到Callable的返回值

使用线程池的方式

背景思路:对于频繁需要创建销毁的的线程,提前创建多个线程,放入线程池中,使用时候取出来,用完放回去.

相关api:ExecutorService 和 Executors

ExcutorService:真正的线程池接口.常见子类ThreadPoolExecutor

method:

void execute(Runnable command) ：执行任务/命令，没有返回值，一般用来执行 Runnable

Future submit(Callable task)：执行任务，有返回值，一般又来执行 Callable

void shutdown() ：关闭连接池

Executors：工具类、线程池的工厂类，用于创建并返回不同类型的线程池

method:

Executors.newCachedThreadPool()：创建一个可根据需要创建新线程的线程池

Executors.newFixedThreadPool(n); 创建一个可重用固定线程数的线程池

Executors.newSingleThreadExecutor() ：创建一个只有一个线程的线程池

Executors.newScheduledThreadPool(n)：创建一个线程池，它可安排在给定延迟后运 行命令或者定期地执行。

线程的同步

多线程可能带来的麻烦**:执行结果的不稳定性**

原因:同样一段代码,会有多个线程执行,这样会面临一种情况:a线程对多条语句只执行了一部分,还没执行完就到了时间片,被阻塞了,此时b线程参与执行了语句.之后a线程恢复,执行未完成部分,但是部分上下文环境已经发生了变化,不能保证当时a线程未被阻塞前执行效果的一致性了.(换句话说,由于b的插手,导致a的运行结果变了)

解决办法:对于有多线程共享的数据的语句,加入同步机制,即一个线程必须执行完这段同步的代码,才能进入阻塞态.

写多线程代码时，需牢记Brian Goetz的同步格言：如果写一个变量，而这个变量接下来可能会被另一个线程读取，或者，如果读一个变量，而这个变量可能已经被另一个线程写入值，那么必须使用同步。

同步机制synchronized,对资源加锁

1.同步代码块

synchronized(类名.class/this){

//需要被同步的代码

}

2.synchronized还可以放在方法声明中，表示整个方法为同步方法，俗称内部对象锁

public synchronized void show (String name){

….

}

同步机制的失效

1.当前线程的同步方法、同步代码块执行结束。

2.当前线程在同步代码块、同步方法中遇到break、return终止了该代码块、 该方法的继续执行。

3.当前线程在同步代码块、同步方法中出现了未处理的Error或Exception，导 致异常结束。

4.当前线程在同步代码块、同步方法中执行了线程对象的wait()方法，当前线 程暂停，并释放锁。

同步机制带来的问题

死锁

Lock(锁)接口,jdk5.0开始

与synchronized不同在

1.Lock机制,需要手动关锁和开锁

2.只有代码块锁,没有方法锁

3.性能更好

注意:在使用锁lock加锁后,如果其他线程也要对加锁代码块访问,识别到已加锁会进入阻塞状态,等到之前锁那段资源访问完毕解锁或者进入wait()后,才会将阻塞状态接触,正式访问加锁资源

在synchronized方法或者代码块中:

注意:以下三个方法只有在synchronized方法或synchronized代码块中才能使用，否则会报 java.lang.IllegalMonitorStateException异常。

wait()方法

作用:令当前线程挂起并放弃CPU、同步资源并等待，使别的线程可访问并修改共享资源，而当 前线程排队等候其他线程调用notify()或notifyAll()方法唤醒，唤醒后等待重新获得对监视器的所有 权后才能继续执行。

如何使用:对象名.wait(),通常直接写wait()

notify()/notifyAll()

作用:唤醒正在排队等待同步资源的线程中优先级最高者结束等待/唤醒正在排队等待资源的所有线程结束等待

如何使用:在当前线程中调用方法： 对象名.notify(),通常直接写notify()

在lock中:

为了线程之间的通信,我们需要通过lock来获取Condition,由Condition来操纵线程之间的通信.

在我们需要的时候,我们就可以调用Condition.await(),去将lock的锁释放,同时该线程会被阻塞;

满足条件时,会调用Condition.signal()/signalAll(),会把之前因为condition阻塞的一个线程激活,前提是调用Condition.signal()/signalAll()的线程必须加锁

应该使用哪种锁？

来自java核心技术中的建议：

1.最好既不使用Lock/Condition也不使用同步方法(synchronized)。在许多情况下，可以使用java.util.concurrent包中的某些机制，它会为你处理所有的锁定。

2.如果synchronized关键字适合你的程序，那么尽量使用这种做法

3.如果特别需要lock/condition结构提供额外的能力，则使用Lock/Condition

volatile字段

用途：将变量声明为volatile后，该变量拥有了内存可见性的语义(semantics for memory visibility),我猜测该变量不会再从寄存器中读取了，每次读取都从内存中读出来。可以保证你的修改，对读取这个变量的所有线程都是可见的。

警告：使用volatile字段的变量并不能提供原子性，即不能保证读取、翻转、写入不被中断。

时,如果不对这段获取数据的时间内解锁的话,线程池的工人线程在执行complete()方法时,会要求加锁lock,但是主线程锁还在,那么必须陷入阻塞等主线程的锁释放,这样效率不高;

对这段获取数据时间解锁,那么两个工人线程就可以利用这段时间去处理数据,那么当主线程处理下一次的rawblob时,很大可能两个工人线程已经完成任务,主线程就不必陷入阻塞,直接取出数据后,就再处理下一个rawblob,充分利用时间.

此外

这段代码中用的是while,而不是if去作为判断,因为有可能主线程想取出的是线程1的数据而陷入了阻塞,但是因为线程唤醒的随机性,有可能随即唤醒的是线程2,那么我们需要的线程1数据并没有被处理好.如果用if,当线程2处理完后,就立即唤醒了主线程,那么就会漏掉我们线程1的数据.用了while后,我们会一直判断blobResult是否处理好,知道我们需要的线程1数据被处理完,才能退出循环,即使主线程会因此多阻塞几次.

Linux命令行选项

以下内容摘自此处

1、原始unix风格
a、命令行选项以连字符’-‘开头，后跟单个字符表示选项，选项后面跟着取值，如：mysql -hlocalhost

b、选项不带取值的，可以组合在一起，如：sed -n -r 可以写成 sed -nr

c、偏爱小写字母，大写字母另有含义，非常简练

2、GNU风格

　a、命令行选项以两个连字符’–’开头，后跟选项名称，选项后面使用空格或者等号 紧接着取值。

b、容易记忆，书写麻烦。

3、大多数程序是两种风格混用，两种风格都支持，前者是后者的缩写方式，如下：

mysql –host localhost 或者 mysql –host=localhost ，缩写方式为mysql -hlocalhost，注意缩写方式尽量不要再写成 mysql -h localhost，或者 mysql -h=localhost，因为这可能会存在解析困难，空格可能会认为是下一个选项，而等号可能会认为是取值的一部分。比如：

mysql -hlocalhost -uroot -p 123456，认为123456 是下一个选项。

mysql -hlocalhost -uroot -p=123456，认为密码是 “=123456”，当然密码错误。

4、通过 xxx –help 我们可以看到命令的用法，会列出选项和选项的缩写方式:

如输入:mysql --help 显示的其中一行为:

-h,　--host=name　　Connect to host.

前面是缩写方式，后面是描述说明。

5、一般的缩写方式是取全称的第一个字符，这就存在冲突的问题:

比如：–host 与–help ，解决办法是：其中一个提供缩写方式，另外一个不提供。当然也可能提供另外一个字符，表示缩写，这个字符是大写形式或者意思相近的字符。比如：mysql 小写p是password的缩写，大写P是port的缩写。

6、对于help，一般会有三种方式，–help、-h、-?

GraphHoperStorage中称为“node”和“edge”的”DataAccess”(存疑)对象的数据布局如下所示：

解释:

1.每一个”node row”成员有:latitude,longitude,和first edgeID(即指向的第一条边的id)

2.每个”edge row”成员有:两个edgeIDs即nextA,nextB,接着 two nodeIDs nodeA and nodeB,还有一些性质如:distance和the flages,….,nodeAid总是小于nodeBid

3.一个node有多条边,这被实现为链表的形式,如:node3第一条边是0-3,接着找3的边从nextB出发找到边1-3,接着从next B出发找到3-5,这时候就得从nodeA出发找3的下一条边于是到了3-7;此时没有下一条边查找中止

4.尽管node4,6没有边,但是他们仍然存在,需要消耗DataAccess的空间;对于OSMReader来说去除很少几点的网络也是不会这样做的

5.edgeref等同于node数据结构出的指针,edge处的nextA,B,换了个名称

以LowLevelAPIExample.java中的一段代码为例,描述上述数据结构的生成过程

在初始化后,我们手动为添加一条edge,nodeA,B的id分别为0,1,即graph.edge(0, 1)方法.在此方法的调用栈内部,首先检测了当前现有edge的总数目是否超标,再检测node数据结构的容量,当新nodeA/B的id>目前nodecount数目后,对于我们线性增加当前node数目到最新id,并且为这些新node分配一个默认edgeref id 值-1.之后接着为edge数据结构内填入nodeA,Bid,再根据nodeA,B的edgeref填入到NextA,B.填入后,再将nodeA,B的edgeref设为该条边的id.

默认情况下edge数据结构还有Dist(距离),Flag(一些tag,标注速度,路况等等复杂信息,由flagencoder来决定),Geo,Name
